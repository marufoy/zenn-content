## ピボットとは
「ピボット」というのはビジネスとか、スポーツとかの用語でも同じようなものがある。
「方向転換」とかそういうニュアンス。
ハッキングにおいては、侵入したマシンを軸に、他のマシンに攻撃を仕掛けることをいう。

試験のネットワークには2種類のゾーンがある。

攻撃者から直接見えるゾーン
攻撃者から直接見えないゾーン

直接見えないゾーンにいるマシンは、Nmapでスキャンしても届かないし攻撃もできない。
でも直接見えるゾーンのマシンを踏み台にすれば、その先のゾーンに手が届くようになる。
これがピボットだ。

# なぜピボットが必要なのか

eJPTの試験環境には複数のネットワークが存在する。

攻撃者から直接アクセスできるネットワークと、
**直接アクセスできないネットワーク**だ。

直接アクセスできないネットワークにいるマシンは
Nmapでスキャンしても届かない。攻撃も当然できない。

しかし試験のフラグはその**奥のネットワークに隠されている**ことがある。

そこで必要になるのがピボットだ。

一度侵入したマシンを**踏み台**にして、
本来届かないはずのネットワークへ侵入する技術である。

要するにこういうことだ。
```
攻撃者 → [マシンA] → [マシンB]
              ↑
          ここを踏み台にする
```

マシンAへの侵入だけでは試験はクリアできない。
その先のマシンBまで到達して初めてフラグが取れる。

**これがeJPTで最も難しいと言われる理由だ。**

例えば、以下の例だと、マシン1からは　2、3しか見えない。
マシン4を見るためには今度はマシン3に入らないといけない。

これが厄介なところ。

![](https://storage.googleapis.com/zenn-user-upload/2bf4b2ca3cee-20260403.png)
eJPTでは複雑な構造でない可能性が高い。**せいぜい2階層ぐらい**までだ。

# ピボットの全体の流れ

## 前提
マシンAへの侵入が完了し、**Meterpreterを取得していること**が前提だ。
まだ通常シェルの場合は先にMeterpreterに昇格すること。
![](https://storage.googleapis.com/zenn-user-upload/14bab8d9dbc3-20260403.png)

## 1. 別のマシンを発見する

マシンAのシェル上で以下のコマンドを打つべし。
自分が今いるマシンがどのネットワークに繋がっているかがわかる。

**IPアドレスの確認**

Windowsの場合
```bash
ipconfig /all
```

Linuxの場合
```bash
ip addr
# または
ifconfig
```

**ルーティングテーブルの確認**

Windowsの場合
```bash
route print
```

Linuxの場合
```bash
netstat -nr
# または
ip route
```

見慣れない別のネットワークアドレスが見えたら
**それがピボット先のセグメント。**


## 2. Meterpreterに昇格する

通常のシェルではピボットができないので注意。
必ずMeterpreterに昇格！
```bash
use post/multi/manage/shell_to_meterpreter
set SESSION <セッション番号>
run
```

---

## 3. addrouteで経路を追加する

Metasploitに「このネットワークへはマシンA経由で行け」と教える作業だ。
```bash
run post/multi/manage/autoroute SUBNET=<ネットワークアドレス> NETMASK=255.255.255.0
```


## 4. portfwdでポートを転送する

### portfwdとは何をしているのか

portfwdは**自分のマシンのポートとターゲットのポートを繋ぐトンネルを作る**作業だ。

例えばマシンBの80番ポートに直接アクセスはできない。
しかし自分のローカルの8080番をマシンBの80番に繋げれば
`localhost:8080`へのアクセスがマシンBの80番に届くようになる。

要するに**「届かない場所への中継地点を作る」**のがportfwdだ。
```bash
portfwd add -l <ローカルポート> -p <ターゲットポート> -r <マシンBのIP>
```

## 5. 転送したポートに攻撃する

あとは転送したローカルポートに対して
普段通りのツールで攻撃するだけだ。
```bash
# スキャン
nmap -sC -sV -p <ローカルポート> 127.0.0.1

# ブルートフォース
hydra -l admin -P /usr/share/wordlists/rockyou.txt 
  ssh://127.0.0.1:<ローカルポート>
```

**ポイントはポートごとに個別にportfwdを張ることである。**
ProxychainやSocksは使わないことが個人的におすすめだ。

## ⚠️ ProxychainとSocksは使うな

### ProxychainとSocksとは何か

ProxychainとSocksは**踏み台マシンを経由して
自分のKaliのツールをそのまま使える**ようにする技術だ。

理論上は非常に強力だ。
portfwdのようにポートを一個一個転送しなくても
ターゲット全域に対して自由にツールが使えるようになる。

「なんだそっちの方が便利じゃないか」と思うだろう。

しかしeJPTでは**罠**だ。

---

### なぜ使わないのか

eJPTの試験環境はブラウザベースの仮想環境だ。
**そもそも通信が不安定**である。

そこにProxychainを噛ませると何が起きるか。

- 処理が重くなりすぎてツールが動かない
- **繋がっているのに繋がっていないと勘違いする**
- **重いだけでまだ処理中なのに失敗だと思い込む**
- 原因不明の沼にハマって時間を溶かす

試験は48時間あるとはいえ、
原因不明の沼は精神的にきつい。

---

### 結論

**素直にportfwdを使うべし。**

手作業でポートを一個ずつ繋いでいく方が
地味でも確実で速い。

これは実際に試験を受けた上での強い推奨だ。
